# 第二步：找争议

## 先看总图

如果把 FAST-LIVO2 放进更大的技术路线里，最容易吵起来的是这 3 件事：

| 争议 | 本质问题 |
| --- | --- |
| 串行更新 vs 联合优化 | 多源观测应该一轮一轮更新，还是放进一个统一大优化？ |
| 轻量地图 vs 丰富地图 | 地图主要服务定位，还是尽量表达完整世界？ |
| Direct vision vs Feature/Learned vision | 视觉前端该直接用像素，还是先提炼成更稳的表示？ |

这一步最重要的不是站队，而是先看清：

**FAST-LIVO2 不是唯一答案，而是一组明确取舍。**

---

## 争议一：串行更新 vs 联合优化

### 问题是什么

LiDAR 和 Vision 进来之后，应该：

- 像 FAST-LIVO2 这样按顺序更新

还是：

- 放进一个统一目标里一起解

### 阵营 A：串行更新更实际

**他们主张**

- 先 IMU 预测
- 再 LiDAR measurement update
- 再 Vision measurement update

**他们为什么这样想**

- 更实时
- 更省算力
- 更容易调试
- 更适合工程实现和板端部署

**FAST-LIVO2 站在哪边**

- 明显站在这一边
- `lio_vio_flg` 明确切分 `LIO` 和 `VIO`
- 执行顺序就是 `Prediction -> LiDAR Update -> Vision Update`

### 阵营 B：联合优化更完整

**他们主张**

- LiDAR 和 Vision 都是观测
- 最好一起放进统一优化里解

**他们为什么这样想**

- 变量相关性保留得更完整
- 不容易受“先更新谁后更新谁”的顺序影响
- 在复杂耦合场景里，理论上更一致

### 这场争议的底层分歧

| 阵营更在意什么 | 内容 |
| --- | --- |
| 工程派 | 实时、稳定、好调、能上板子 |
| 统一建模派 | 一致性、完整性、相关性保留 |

### 和第一步的关系

它直接对应第一步里的：

- `IMU 先预测`
- `LiDAR 提供观测`
- `Vision 也提供观测`

---

## 争议二：轻量地图 vs 丰富地图

### 问题是什么

地图到底应该主要做什么？

- 只是服务定位和更新

还是：

- 也尽量承载丰富几何、纹理、长期世界表达

### 阵营 A：轻量几何地图更实用

**他们主张**

- voxel + plane 这种结构就够好
- 地图先服务状态估计

**他们为什么这样想**

- 查询快
- 更新快
- 内存更可控
- 对实时定位来说，够稳比够华丽更重要

**FAST-LIVO2 站在哪边**

- 更偏这一边
- `BuildVoxelMap`
- `UpdateVoxelMap`
- 视觉部分也尽量挂在已有空间结构上

### 阵营 B：地图太轻会丢信息

**他们主张**

- 只存 plane/voxel 太保守
- 会丢掉很多对重建和视觉都重要的信息

**他们为什么这样想**

- 世界不是处处平面
- 纹理和细节也重要
- 如果目标不只是定位，rich map 更有价值

### 这场争议的底层分歧

| 阵营更在意什么 | 内容 |
| --- | --- |
| 工作内存派 | 地图是给优化服务的 |
| 世界模型派 | 地图应该尽量表达真实世界 |

### 和第一步的关系

它直接对应第一步里的：

- `地图不是最后结果，而是工作内存`
- `地图持续参与下一轮`

---

## 争议三：Direct vision vs Feature/Learned vision

### 问题是什么

视觉前端到底该不该继续坚持 direct method？

### 阵营 A：direct method 更贴合 FAST-LIVO2

**他们主张**

- 直接利用 patch 光度误差做观测更新

**他们为什么这样想**

- 少一层中间抽象
- 更贴近原始观测
- 更容易和 LiDAR 几何先验结合
- 很适合“先有预测，再局部修正”的流程

**FAST-LIVO2 站在哪边**

- 明显站在这一边
- `VIOManager::processFrame`
- `computeJacobianAndUpdateEKF`
- `generateVisualMapPoints`

### 阵营 B：direct method 太脆

**他们主张**

- 直接法容易受曝光、模糊、遮挡、动态物体影响
- 更稳的方案可能是 feature-based 或 learned front-end

**他们为什么这样想**

- patch 光度一致性假设很强
- 真实世界并不总满足这个假设
- 为了补这个问题，系统常常要加很多补丁

### 一个重要信号

如果一个系统里有很多这种开关：

- `normal_en`
- `raycast_en`
- `inverse_composition_en`
- `exposure_estimate_en`

这通常说明：

**核心路线有效，但具体怎么做并没有完全定论。**

### 这场争议的底层分歧

| 阵营更在意什么 | 内容 |
| --- | --- |
| 直接观测派 | 少做抽象，少丢信息 |
| 稳健表示派 | 原始像素太脆，先提炼再估计更稳 |

### 和第一步的关系

它直接对应第一步里的：

- `Vision 也提供观测`
- `Map 提供视觉参考`
- `预测 -> 观测 -> 更新`

---

## FAST-LIVO2 的立场，一眼看懂版

| 问题 | FAST-LIVO2 更偏哪边 |
| --- | --- |
| 更新方式 | 串行更新 |
| 地图表达 | 轻量几何地图 |
| 视觉路线 | Direct method |

所以你可以把它记成一句话：

**FAST-LIVO2 不是在所有方向都取最重最全的表达，而是在实时系统里选择了一套偏工程可运行的组合。**

---

## 速记版

如果只留 3 句：

1. 它更相信 sequential update，而不是统一大优化。
2. 它更相信“够用而稳”的地图，而不是更重的世界模型。
3. 它更相信 direct vision，但也因此必须面对 direct vision 的脆弱性。
